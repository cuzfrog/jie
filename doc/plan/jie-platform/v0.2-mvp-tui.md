# jie-platform 0.2 — TUI MVP

The v0.2 deliverable: a working TUI in `packages/jie-tui/` that replaces the v1 stub (`throw "TUI not implemented..."`). Drives the full team interactively. Spec: `ui/tui.md`. AC: `00-tui-user-scenarios.md` (T1–T5).

## Scope

**In scope (v0.2):**

- All T1–T5 scenarios from `00-tui-user-scenarios.md`.
- Multi-team in the TUI (`loadTeam`, `/team` slash command, swap-back event buffer) — per `ui/tui.md` "Multi-team".
- Full flag parity with `jie -p` (`--team`, `--api-key`, `--resume`, `--continue`) — per `ui/tui.md` "Flag parity".
- Library: `@earendil-works/pi-tui` (sister package to `pi-ai` / `pi-agent-core`; both already vendored).
- Hand-recorded JSONL fixtures under `tests/e2e/tui/fixtures/`; replay-based TUI tests under `tests/e2e/tui/`.

**Out of scope (v0.2 stays Day 2+):**

- The jie-team dev team (`specs/jie-team/`). v0.2 TUI runs against any user-installed team, but the dev team itself is not in v0.2.
- MCP client (`specs/jie-platform/10-configuration.md` "MCP Server Configuration"). Archived scenario 7 stays archived.
- jie npm publish / install script / `jie upgrade` (backlog #20).
- TUI a11y / theme customization.
- Cross-process team visibility.
- vim-style keybindings (Day 2+ add-on).
- Hand-rolled raw-mode renderer / Ink / blessed (decision: pi-tui; the TUI does not own the rendering layer).
- LLM-driven recording of event traces (Day 2+; v0.2 fixtures are hand-recorded).

## Phases

### Phase 1: render primitives + input loop (T1 smoke)

- Replace the v1 stub with a real `startTUI` that:
  - Checks `process.stdin.isTTY`; logs the non-TTY error and returns non-zero otherwise.
  - Constructs `ProcessTerminal` and `TUI` from `pi-tui`.
  - Mounts a `Container` with `StatusBar`, `AgentsRail`, `ChatPane`, `Editor`, `Footer` — minimal, hard-coded content ("hello agent").
  - Listens for `Ctrl+D`; restores TTY; exits 0.
- AC: `jie` opens a TUI; the layout matches the figure in `ui/tui.md` "Layout"; `Ctrl+D` exits 0.
- Test: `tests/e2e/tui/boot.test.ts` drives a fake `VirtualTerminal`, asserts the boot frame and the exit on `Ctrl+D`.

### Phase 2: state derivation (reducer)

- Define `TuiState` in `packages/jie-tui/state.ts`:
  ```typescript
  interface TuiState {
    teamId: string | null;
    agents: Map<string, AgentUiState>;
    focusedAgentKey: string | null;
    queue: string[];
    inputBuffer: string;
    inputHistory: string[];
    historyIndex: number;
    errorBanner: string | null;
  }
  ```
- Define `reduce(state: TuiState, envelope: EventEnvelope): TuiState` — pure. Subscribes to:
  - `team.{id}.loaded` → seed `agents` from `payload.agents`.
  - `agent.turn.start` → set agent status `busy`.
  - `agent.idle` → set agent status `idle`; clear `errorBanner` if set.
  - `agent.stream.chunk` → append to `agents[focused].currentTurn.text`.
  - `agent.stream.end` → finalize the current turn.
  - `agent.tool.call` → push a tool card to `agents[focused].history`.
  - `agent.tool.result` → push the result card.
  - `agent.queue.update` → replace `queue` with `payload.prompts`.
  - Domain events (`custom.{team_id}.{topic}`) → push a timeline entry.
- AC: `reduce` is a referentially-transparent function. Test it against the T1 trace.

### Phase 3: layout (StatusBar, AgentsRail, ChatPane, Footer)

- Implement each pane as a `Component` subclass with `render(width: number): string[]`.
- StatusBar reads `state.teamId`, `state.focusedAgent`, `state.queue.length`, `state.agents[focused].status`.
- AgentsRail reads `state.agents`; pins leader to top with `★`; renders `<role>  <status>` per row.
- ChatPane reads the focused agent's `history` + `currentTurn`; renders the chat history scrolling upward (newest at bottom); inserts a `▌ thinking…` line on `block_type: "thinking"`; inserts tool cards inline.
- Footer reads the keymap table; renders the legend.
- AC: T1's first frame matches the figure in `ui/tui.md` "Layout". Snapshot file: `tests/e2e/tui/__snapshots__/t1-boot.snap`.

### Phase 4: slash commands

- `/login` — open a `SelectList` of providers from `pi-ai`'s `getProviders()`; on select, open a hidden `Input` for the API key; write to `~/.jie/auth.json` (mode `0600`).
- `/logout [<id>]` — clear one or all entries.
- `/model <p>/<m>` — validate and write to `~/.jie/settings.json`.
- `/team [<id>]` — open the team picker (`SelectList` with `fuzzyFilter` over `teamRegistry.listInstalled()`); on select, call `handle.loadTeam(id)`. With a positional `<id>`, validate and call `loadTeam` directly.
- `/clear` — clear `state.agents[*].history` and `state.agents[*].currentTurn`. Memory rows on disk are untouched.
- `/help` — open an overlay rendering the keymap.
- `/exit` — same as `Ctrl+D`.
- AC: T4's first-time setup flow works end-to-end.

### Phase 5: streaming + tool cards + thinking

- Append `agent.stream.chunk` content to `state.agents[focused].currentTurn.text` (no per-chunk re-render of the chat history above).
- Tool cards: render the call as `● <name>  <input>` on `agent.tool.call`; render the result as `✓ <name>  <ms>ms` (or `✗ <name>  <error>`) on `agent.tool.result`. Track `cardExpanded` per card; `Space` toggles in the focused card.
- Thinking: render `▌ thinking…` on the first `block_type: "thinking"` chunk; track `blockExpanded` per block; `T` toggles. Default collapsed.
- AC: T2's `manager → worker` flow renders the tool cards inline; T5's queued-prompt indicator renders the queue depth.

### Phase 6: snapshot / replay tests (T1–T5)

- For each of T1–T5, ship:
  - `tests/e2e/tui/fixtures/t<n>.jsonl` — a hand-recorded `EventEnvelope` trace.
  - `tests/e2e/tui/t<n>.test.ts` — runs the trace through the reducer, then through the renderer on a `VirtualTerminal`, asserts the final screen frame against `__snapshots__/t<n>.snap`.
- The fixtures are **hand-recorded** in this phase; the recorder (re-using the e2e infrastructure) is Day 2+.
- AC: all five T1–T5 tests pass; CI runs them on every push and PR.

### Phase 7: docs + closeout

- Update `README.md` and `doc/DEVELOPMENT.md` to point at the TUI.
- Refine the v0.2 doc set: cross-check `ui/tui.md` against `00-tui-user-scenarios.md`; ensure ADR 25 and ADR 19 are consistent.
- Update `MILESTONES.md` — close out the v0.2 stage3 and stage4 checkboxes; bump the v0.2 release notes.
- Open a v0.2 stage5 cleanup PR (refactor + dead code removal — the TUI's reducer and renderer will leave scratch code in their wake).

## Functional AC

`doc/specs/jie-platform/00-tui-user-scenarios.md` (T1–T5).

Plus the cross-cutting AC from `ui/tui.md`:

- **Non-TTY refuses.** `jie` (no flags) under a piped stdout logs the non-TTY error and exits non-zero.
- **Ctrl+C interrupts a turn.** Mid-turn `Esc Esc` (or `Ctrl+C`) publishes a synthetic interrupt; the in-flight LLM call's `AbortSignal` fires; the TUI does not exit.
- **Ctrl+D exits cleanly.** With a turn in flight, prompts for confirmation; without one, exits 0 directly.
- **Snapshot tests run in CI.** All five T1–T5 JSONL traces replay; the rendered frames match the snapshots.

## E2E test backend

Same as v1: the TUI scenarios (T1–T5) replay recorded traces, not live LLM calls. The CI backend (NVIDIA Nemotron) is used by the v1 `tests/e2e/v1-scenarios.test.ts` (not the TUI tests). The TUI tests are deterministic by construction.
